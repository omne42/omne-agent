# v0.2.0：Codex 功能对齐 + 跨项目精华 TODO

> 目标：**先把 Codex 的“可被编排的 Agent CLI”完整做出来**（协议/事件/执行/沙箱/审批/工具/skills/MCP），再在此之上叠加 CodePM 的并发编排与交付适配（Git/远程等都只是 adapter）。
>
> 硬约束：**v0.2.0 不使用 Docker**（也不把 Docker 当作可选后端）。
>
> 原则：先把数据结构与边界做对；“功能对齐”不是复制粘贴，而是把设计抽象成更干净、更可测、更可维护的 Rust 系统，并把其它项目的优点落到接口与 TODO 里。

## E. 角色 / 模式 / 工作流（必须从 Day 1 落地）

> 来源：Kilo（modes/workflows/skills）、Claude Code（plugins/commands/hooks/allowed-tools）、OpenCode（skill/plugin/permission）。

- Mode（角色）= 权限边界（Kilo 的 `.kilocodemodes` 思路）
  - [ ] 角色定义：Architect/Coder/Reviewer/Merger/Builder/Debugger（最小集）
  - [ ] `read/browser/command/edit(fileRegex)` 这类权限组（可版本化、可分享）
  - [ ] “按角色的 allowed-tools/execpolicy”落盘（不要靠 prompt）
- Workflow / Commands（可版本化 spec）
  - [ ] 命令即工作流：markdown + frontmatter（Claude Code 思路：`allowed-tools` + 注入 `git diff`/`status`）
  - [ ] Orchestrator 读取 workflow spec → 生成并发子任务（Kilo workflow 拆分范式）
  - [ ] 子任务并行执行 + 汇总 review（多子 agent 并行）
- Hooks（Claude Code 的 hook 体系）
  - [ ] `SessionStart/PreToolUse/PostToolUse/Stop` 等事件点（先做最小子集）
  - [ ] hook 能注入 additionalContext（安全提醒/约束/环境提示），并可落盘审计
- “特殊指令”（OpenCode 风格）
  - [ ] 约定一套最小的 slash/at 指令（例如 `@file`/`@diff`/`/plan`/`/approve`），并在 app-server 协议里结构化表达（不要只靠文本解析）

## F. Provider / Model 路由（必须学 CCR + OpenCode）

> 来源：Claude Code Router（routing + transformers）、OpenCode（provider-agnostic & config）。

- Provider 抽象（OpenCode 的 provider-agnostic）
  - [ ] `Provider` trait + capability flags（tools/vision/reasoning/json-schema/streaming）
  - [ ] provider 配置与密钥加载（不进日志、可预设/可分享）
- Router（CCR 的场景路由 + subagent routing）
  - [ ] role→model 路由：Architect/Reviewer 用强推理；Coder 用便宜/快；长上下文单独路由
  - [ ] keyword/标签触发路由（“依据关键词自动切模型”）
  - [ ] subagent 指定模型（子任务头部或结构化字段强制路由）
- Preset/Config（CCR 的“可迁移配置”产品化）
  - [ ] preset 导入/导出（导出时对敏感字段脱敏占位符）
  - [ ] config layering 与“为什么生效的是这个值”的可解释性（Codex 的 config_layer_stack 思路）
- Transformers（未来多 provider 的变换槽位）
  - [ ] 先把内部“统一请求/响应结构”钉死，provider 适配集中在一层（别把差异散落到业务里）

## H. 多任务/多子 agent 并发（RTS 的骨架，不是锦上添花）

> 来源：Codex（thread fork/turn interrupt）、Claude Code（并行专项 agents）、Kilo（workflow 并行拆分）、OpenCode（scheduler）。

- Scheduler
  - [ ] fan-out：把一个 turn/workflow 拆成多个子 agent/子任务并行执行
  - [ ] fan-in：把子任务结果结构化汇总（带 provenance：引用哪些 artifacts/items）
  - [ ] 限流与公平性：worker pool + 优先级（Attention 优先、后台次之）
- 长任务
  - [ ] heartbeat/idle detection（“没输出也没结束”的卡死必须可见）
  - [ ] 可中断恢复：interrupt 后允许继续 turn（而不是只能重来）
