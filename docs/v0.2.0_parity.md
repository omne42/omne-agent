# v0.2.0：Codex 功能对齐 + 跨项目精华 TODO

> 目标：**先把 Codex 的“可被编排的 Agent CLI”完整做出来**（协议/事件/执行/沙箱/审批/工具/skills/MCP），再在此之上叠加 CodePM 的并发编排与交付适配（Git/远程等都只是 adapter）。
>
> 硬约束：**git/workspace 不以 Docker 为前提**（我们用 `/tmp`/worktree 等目录隔离；不禁止 agent 自己运行 Docker/Podman）。
>
> 原则：先把数据结构与边界做对；“功能对齐”不是复制粘贴，而是把设计抽象成更干净、更可测、更可维护的 Rust 系统，并把其它项目的优点落到接口与 TODO 里。

## 0) 已决定的约束（写死，别再争论）

- **Resume 语义（Codex 风格）**：一个 session 必须包含完整上下文：对话、tool/进程步骤、中间态产物、审批与结果；`resume` 时用这些历史拼合上下文并继续对话。**不承诺“原进程继续跑”**，但必须承诺“历史不丢、可继续推进”。
- **Artifact 定义**：repo/workspace 里的代码变更属于“repo 更新”，不叫 artifact。artifact = **给用户看的文档** + **不进 repo 的临时产物**（由 agent 判断生成/归档），参考 `~/.gemini/antigravity/brain/*/*.md + *.metadata.json` 的形态。
- **进程 attach**：只允许 `inspect/tail/follow`（只读查看）或 `kill`，**禁止交互输入**（不做 stdin/PTY 交互）。
- **模型路由优先级（已定）**：`subagent 强制 > project override > keyword rule > role default > global default`，且历史记录必须能看到“本次用了什么模型、为什么”。
- **跨 session 记忆（可选）**：默认不启用；未来只记录“用户特征/偏好/擅长领域/常用库/代码风格”等 profile，不默认记代码内容。

## B. 核心语义（必做）

- `Thread/Turn/Item` 强类型事件模型
  - 已定：**append-only 落盘事件**是权威来源；流式通知只是优化，断线/lag 可用 `resume + 从 offset/seq 重放` 补齐。
  - 已定：每个 `Item`/`ProcessEvent` 必须有**单调递增序号**（同一 thread 内）用于重放/去重/拼合历史。
  - 已定：订阅可靠性（最小语义）：
    - 客户端订阅使用 `since_seq`（从 `since_seq + 1` 重放/继续），并持久化自己的 `last_seq`（掉线后能续）。
    - 服务端按 `seq` 有序推送；允许重复（at-least-once），客户端按 `seq` 去重即可。
    - 不用 ack 作为正确性前提：**不丢的保证来自落盘 log**，订阅只是投递优化；消费端 lag 了就重连补齐。
  - [ ] Item 覆盖：message/reasoning/tool/file edit/diff/approval
  - [x] 落盘：JSONL rollout + 回放（replay）+ 最小索引（list/show）
  - [x] 派生视图：Attention/Inbox（v0.2.0 MVP：NeedApproval/Running/Failed/Stuck/Done/Idle/Archived）
  - [ ] 长任务防失控：loop/cycle detection、超时、自动 compact（把“烧钱卡死”变成可见状态）
  - [ ] checkpoint/回滚（turn 级别，至少支持回到某个稳定点）
  - [x] **中间态 artifacts（必须）**：stdout/stderr（process logs）与 user artifacts 采用追加/版本化写入，并支持随时读取/清理
  - [x] **运行中可观测（必须）**：运行中的后台进程支持 `list/inspect/tail/follow/kill`；tool events 全量落盘可回放

## H. 多任务/多子 agent 并发（RTS 的骨架，不是锦上添花）

> 来源：Codex（thread fork/turn interrupt）、Claude Code（并行专项 agents）、Kilo（workflow 并行拆分）、OpenCode（scheduler）。

- Scheduler
  - 已定：任务分为两类：
    - **前台阻塞任务（foreground）**：需要串行/需要人盯的关键路径（默认不并发）。
    - **后台任务（background）**：用于多并发吞吐（用户可随时 inspect/kill；完成后进入 Attention/通知）。
  - [ ] fan-out：把一个 turn/workflow 拆成多个子 agent/子任务并行执行
  - [ ] fan-in：把子任务结果结构化汇总（带 provenance：引用哪些 artifacts/items）
  - [ ] 限流与公平性：worker pool + 优先级（Attention 优先、后台次之）
- 长任务
  - [ ] heartbeat/idle detection（“没输出也没结束”的卡死必须可见）
  - [ ] 可中断恢复：interrupt 后允许继续 turn（而不是只能重来）
  - [ ] 预算：token/时间/tool 调用次数上限；超预算进入 `Stuck` 并要求人工决策（已实现 step/tool/time 上限，支持 `CODE_PM_AGENT_MAX_*` 覆盖；仍缺 token budget + 解释性 artifacts）

  - [ ] “提前返回”策略：若后台任务全部提前完成则立即收口；否则输出进度摘要并持续更新（直到完成或进入 Stuck/需要人介入）
