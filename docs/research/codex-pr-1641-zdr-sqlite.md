# Codex PR #1641：ZDR 与本地敏感数据存储（sqlite + 内存密钥）

> Source: `https://github.com/openai/codex/pull/1641`
>
> Merged: **2025-07-09**
>
> Accessed: **2026-01-28**

---

## 1) 这个变更是为了什么？

PR #1641 的核心诉求是：**让 Codex CLI 在“Zero Data Retention (ZDR)”语境下更安全地处理本地敏感数据**。

它指出 Codex 之前会把工具输出（tool outputs）等内容持久化到 `.codex` 目录，这可能包含敏感信息（文件内容、命令输出、路径、日志片段等）。PR 的方向是：

- 把 **tool output + 其它敏感内容** 不再直接明文落盘到 `.codex`。
- 改为写入一个 **sqlite 数据库**，并且：
  - 数据以 **加密形式** 存储；
  - **加密密钥只在内存中存在**（不落盘）。
- 当启用 `disable_response_storage` 且使用 **ZDR key** 时：
  - sqlite db 放在 `/tmp`；
  - 进程退出时删除。

---

## 2) 好处（为什么这是个“正确方向”）

- **降低明文落盘风险**：工具输出往往是最敏感的部分（读取文件、跑命令、diff 结果），把它们从可读 JSONL/文件迁移到加密存储，泄露面显著下降。
- **更贴近 ZDR 的预期**：ZDR 只解决“服务端不留”，但客户端仍可能在本地留痕；这条 PR 补齐了“本地留痕最小化/不可恢复”的一部分。
- **运行期可用、退出后不可解密（默认）**：密钥仅在内存中，进程停止后即使 sqlite 文件残留也很难复原内容（取决于实现细节与攻击模型）。

---

## 3) 坏处 / 风险（你需要知道的 tradeoffs）

- **复杂度上升**：引入加密与数据库层，意味着更多边界情况（崩溃恢复、并发、锁、写放大、性能退化）。
- **“删除”不是“消失”**：即便放 `/tmp` 并删除：
  - 仍可能被 OS/文件系统/快照/备份保留一段时间；
  - sqlite 的 journaling/WAL、磁盘缓存、swap、core dump 都可能带来残留风险（视实现与系统配置而定）。
- **可观测性与可调试性下降**：明文日志/事件回放更难，需要额外的“安全调试开关”或脱敏视图。
- **安全边界依赖攻击模型**：密钥在内存中仍可能被高权限进程、调试器、恶意扩展、内存转储获取；这并非“绝对安全”，而是“减少常见落盘泄露面”。

---

## 4) [Image #1] 的说法对吗？

如果图里是在总结“PR #1641 通过把 tool output 等敏感数据存到加密 sqlite（密钥只在内存），并在 ZDR+disable_response_storage 时把 db 放 /tmp 且退出删除”——**整体是对的**。

需要补充的点通常有两类：

- **/tmp 并不等于安全**：它更多是“生命周期更短/更少被备份”的经验做法，但依然可能残留（Linux 上 `/tmp` 也可能被保留一段时间，这在 PR review 里也被提到）。
- **这不是万能的 ZDR**：它解决的是“客户端本地落盘泄露面”之一；仍要考虑内存泄露、日志、swap、崩溃转储等。

---

## 5) 对我们当前目录（CodePM 这组仓库）的直接启发

对 `codex_pm/`（CodePM app-server）来说，最关键的不是 sqlite 这个实现细节，而是一个原则：

- **如果你为了对齐 Responses 的“unrolled stateless loop / compaction / encrypted_content round-trip”而必须保存 raw items，那么就必须重新评估“本地存储敏感数据”的边界**。

我们现在的默认策略是：

- `ThreadEvent` 作为审计/回放记录是 **append-only 且会脱敏**（安全优先）。
- 若要“与 Codex CLI 完全对齐”的 Responses 路线，就需要一个 **raw history** 通道（不依赖 redacted events 来重建 prompt）。

这正是 PR #1641 要解决的矛盾：**既要保留足够精确的上下文（包括 tool outputs），又要把落盘风险降到可接受水平**。
