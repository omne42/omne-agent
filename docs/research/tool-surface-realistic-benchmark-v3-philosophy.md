# Tool Surface Realistic Benchmark v3：测试经验与测试哲学

## 1. 我们在测什么

我们测的是 **Agent 系统在真实任务里的可执行可靠性**，不是模型会不会“说漂亮话”。

关键点：

- 用户提示词必须像真实用户，不应携带流程脚本或格式绑架。
- 工具调用成功要靠运行时证据判定，不靠模型自报 `success:true`。
- 需要把“会偷懒不调工具直接结案”的行为当作一类核心风险来治理。

## 2. v3 的核心改进

相对早期基准，v3 明确做了三件事：

1. 去掉 Prompt 污染：保留自然语言任务意图，不把恢复策略写进用户提示。
2. 强化错误注入：工具返回包含 `suggested_args` + `instruction`，明确要求重试而不是结束。
3. 增加无工具结案拦截：当模型在该类任务中零工具直接结案，系统注入一次纠正消息，强制再试一轮。

并行地，业务侧 `mcp_call` schema 强化为：

- `arguments` 必填；
- `arguments` 必须是嵌套 object；
- 明确禁止把 MCP 参数拍平到根层。

## 3. v3 smoke 结果（codex_responses）

运行条件：

- modes: `direct + recovery`
- cases: 10（共 20 行结果）
- tool surface: 43

结果摘要（`2026-03-03` 这轮）：

- `pass_rate = 100%`
- `skip_tool_rate = 0%`
- `target_miss_rate = 0%`
- `fake_success_rate = 0%`
- `guard_trigger_rate = 5%`（20 行里触发 1 次）

这说明：

- 去掉污染后，模型仍可能偶发“先口头结案”；拦截器是必要的防线；
- 加了拦截与强约束后，最终执行链路可收敛到稳定通过。

## 4. 我们确认过的反模式（必须避免）

1. 把框架职责写进用户 Prompt（例如“当 ok=true 输出固定 JSON”）。
2. 用输出格式当通过条件，替代真实工具执行结果。
3. 强制单工具注入，导致结果脱离真实多工具竞争环境。
4. 只看最终文本，不看中间 step 的工具行为和参数修复过程。

## 5. 评测哲学（长期执行）

1. **真实性优先于可控性**：宁可指标更难看，也不能靠提示词“调教”出虚高分。
2. **执行证据优先于文本证据**：运行时事件是主证据，模型文本只是辅证据。
3. **系统修复优先于提示词修复**：防幻觉、防偷懒应落在引擎层，不应依赖 prompt 技巧。
4. **失败可追责**：每次失败都要能回放到具体 step、参数和工具返回。
5. **指标要反映业务风险**：`skip_tool`、`target_miss`、`fake_success` 比“说得像不像”更重要。
6. **随机性可接受，但边界要收敛**：允许单轮波动，但长期统计必须在可控范围内。

## 6. v3 之后的执行规范（建议固定）

每次回归至少产出：

- `report.md`（总览）
- `raw_results.json`（可审计原始链路）
- `summary.easy.md`（快速读）
- 失败样例的逐条归因（若有）

每次改动至少复核：

- `skip_tool_rate` 是否上升
- `target_miss_rate` 是否上升
- `fake_success_rate` 是否上升
- `mcp_call` 嵌套参数失败率是否回升

## 7. 一句话原则

**Benchmark 不是为了证明模型聪明，而是为了证明系统在真实任务里不作弊、可追责、能收敛。**
