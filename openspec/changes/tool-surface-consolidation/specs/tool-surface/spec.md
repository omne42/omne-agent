# 规格增量：tool-surface（聚合工具与渐进披露）

## 新增要求

### 要求：默认模型工具面必须收敛到聚合入口

系统在默认配置下，模型可见工具必须收敛为不超过 5 个聚合工具，且不直接暴露 legacy 细粒度工具集合。

#### 场景：默认工具暴露

- 给定系统使用默认工具配置
- 当开始一次 turn 并构建模型工具 schema
- 则模型可见工具数量不超过 5 个
- 且工具集合为 `workspace/process/thread/artifact` 与可选 `integration`

### 要求：help-first 渐进披露

每个聚合工具必须支持统一帮助入口，默认仅提供最简可用法，进阶参数通过 help 显式请求。

#### 场景：查询最简帮助

- 给定调用 `{"op":"help"}`
- 当聚合工具处理请求
- 则返回 `quickstart`（最短示例）与 `advanced`（扩展参数）
- 且不执行任何副作用动作

#### 场景：按主题查询帮助

- 给定调用 `{"op":"help","topic":"read"}`
- 当聚合工具处理请求
- 则只返回对应主题的帮助内容
- 且包含参数约束与错误示例

### 要求：Facade 错误码稳定

聚合入口必须返回稳定错误码，保证前端/日志可可靠识别错误类型。

#### 场景：参数错误返回稳定错误码

- 给定聚合工具调用缺少必要参数或参数类型错误
- 当聚合工具解析请求失败
- 则返回错误码 `facade_invalid_params`
- 且错误响应中包含 `facade_tool` 与可选 `op`

#### 场景：动作不支持返回稳定错误码

- 给定聚合工具调用 `op` 不受支持
- 当聚合工具尝试路由
- 则返回错误码 `facade_unsupported_op`
- 且错误响应中包含 `facade_tool` 与 `op`

### 要求：mode 与 role 共同生效且职责分离

系统必须同时支持 mode 与 role 两个维度，并在工具面与执行面分别生效。

#### 场景：role 影响模型工具暴露

- 给定线程 role 为 `chatter`
- 当系统构建模型工具 schema
- 则应隐藏高风险变更类工具暴露（例如写文件/启动进程）
- 且只保留对应角色允许的工具子集

#### 场景：mode 仍决定运行时权限链

- 给定线程 mode 为受限模式且 role 为高权限角色
- 当工具调用进入执行链
- 则最终决策仍必须经过 mode 的拒绝语义
- 且不得因 role 提升而绕过 mode/approval/execpolicy 的拒绝

## 变更要求

### 要求：聚合入口不得绕过细粒度安全链路

聚合工具在执行非 help 操作时，必须先映射到明确内部 action，并沿用现有权限/审批链路。

#### 场景：映射后动作受 allowed_tools 约束

- 给定线程配置 `allowed_tools` 不包含目标内部 action
- 当聚合工具路由到该 action
- 则调用被硬拒绝
- 且拒绝结果与 legacy 工具路径一致

#### 场景：映射后动作受 mode/execpolicy/approval 约束

- 给定映射动作为命令执行类
- 当触发 `process/start` 路径
- 则必须继续经过 `mode -> sandbox -> execpolicy -> approval`
- 且拒绝/审批语义保持不变

### 要求：可审计映射信息

工具执行事件必须可回放地标识 facade 与映射后的内部 action。

#### 场景：事件可追踪映射

- 给定一次聚合工具调用成功或失败
- 当写入 `ToolStarted/ToolCompleted`
- 则事件中可定位 `facade_tool`、`op`、`mapped_action`
- 且可据此复原真实权限决策路径
